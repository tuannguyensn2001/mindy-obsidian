# Engineering practice

## When to use this skill

Use this skill when the source mainly communicates best practices, heuristics, guidelines, anti-patterns, or operational habits.

Typical signals:
- The source recommends how engineers should work, review, test, debug, deploy, or maintain systems.
- The content includes rules of thumb, recurring mistakes, or practical guidance.
- The reader needs applied guidance, not deep internal explanation.

Good fit examples:
- Code review best practices
- Logging and observability guidelines
- How to write safe migrations
- Common API design pitfalls
## Rewrite instructions
- Focus on the principle, heuristic, or practice being recommended.
- Explain why it matters, not just what to do.
- Clarify when it applies and when it may not apply.
- Highlight common mistakes, anti-patterns, or failure modes if present in the source.
- Turn vague guidance into more operational notes without inventing details.
- Use this structure when relevant:
	- Overview
	- Core practices
	- Why they matter
	- When to apply them
	- Common pitfalls
	- Practical takeaways
	- Gaps / ambiguity
- Do not present rules without context if the source includes rationale.
