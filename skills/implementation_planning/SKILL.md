# Implementation planning

## When to use this skill

Use this skill when the source is mainly about what to build, how to execute, or how to roll something out.

Typical signals:
- The content includes steps, milestones, dependencies, rollout ideas, or validation plans.
- The source is action-oriented rather than concept-oriented.
- The reader needs a clearer execution plan from messy technical notes.

Good fit examples:
- Planning a migration from monolith to services
- Implementing a new auth flow
- Rolling out feature flags
- Planning a data backfill or schema change
## Rewrite instructions

- Focus on what needs to be built, changed, validated, migrated, or shipped.
- Extract the implementation goal, scope, and expected outcome.
- Rewrite steps into a clearer execution order when appropriate.
- Identify dependencies, prerequisites, unknowns, and validation points.
- Highlight risks, assumptions, and likely decision points during implementation.
- If the source supports it, surface checkpoints, milestones, or success criteria.
- Use this structure when relevant:
	- Overview
	- Goal and scope
	- Proposed approach
	- Execution steps
	- Dependencies and risks
	- Validation / success criteria
	- Gaps / ambiguity
- Prioritize actionability and execution clarity.
